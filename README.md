# REAL OTD Claims Automation Tool

This tool automates the collection and claiming of historical earnings (OTD claims) on the Real app. It fetches available card claims for each configured account and automatically claims the top 2 cards per sport to maximize earnings.

## Setup

### 1. Create Virtual Environment (macOS)

```bash
# Clone the repository
git clone https://github.com/raiyanxk/opt1ms3r.btc.git

# Navigate to the project directory
cd opt1ms3r.btc

# Create a virtual environment
python3 -m venv venv

# Activate the virtual environment
source venv/bin/activate
```

### 2. Install Dependencies

```bash
pip install -r requirements.txt
```

### 3. Configure Environment Variables

Create a `.env` file in the project root with your accounts' auth info:

```
ACCOUNT_1=auth_info_1
ACCOUNT_2=auth_info_2
...
```

### 4. Update `AUTH_INFOS` in the scripts
**Update `AUTH_INFOS` in `get_claims.py`**:
   ```python
    AUTH_INFOS = {
       "ACCOUNT_1": os.getenv("ACCOUNT_1"),
       "ACCOUNT_2": os.getenv("ACCOUNT_2"),
        ...
    }
   ```

**Update `AUTH_INFOS` in `do_claims.py`**:
   ```python
    AUTH_INFOS = {
       "ACCOUNT_1": os.getenv("ACCOUNT_1"),
       "ACCOUNT_2": os.getenv("ACCOUNT_2"),
        ...
    }
   ```

## Usage

### Step 1: Fetch Claims Data

Run `get_claims.py` to fetch the latest historical earnings data from the REAL API:

```bash
python get_claims.py
```

This creates JSON files for each account in the `claims/` directory. For example on 2026-05-16, you would see:
- `ACCOUNT_1_claims_2026-05-16.json`
- `ACCOUNT_2_claims_2026-05-16.json`
- ...

### Step 2: Claim Cards

Run `do_claims.py` to automatically claim the top 2 cards per sport:

```bash
python do_claims.py
```

The script will:
1. Read the claims JSON files
2. Filter out disabled cards
3. Claim up to 2 cards per sport (respecting the daily claim limit)
4. Print success/failure messages with error details

## Token Generation

The REAL API requires two types of authentication tokens generated by `token_generation.py`:

### 1. Native Request Token
- **Format**: `timestamp.nonce.signature`
- **Generation**:
  - Timestamp (in milliseconds): Current Unix time × 1000
  - Nonce: First 8 characters of a UUID
  - Message: `timestamp.nonce`
  - Signature: HMAC-SHA256 of message using a secret key, base64 encoded (first 16 bytes only)
- **Purpose**: Cryptographically signs the request for server validation

### 2. Request Token (Hashids)
- **Format**: Hashids-encoded timestamp
- **Generation**:
  - Encode the timestamp using Hashids with a salt and minimum length
  - Produces a compact string representation of the timestamp
- **Purpose**: Provides additional obfuscation of request timing

Both tokens are included in API request headers:
- `real-native-request-token`: Native request token
- `real-request-token`: Hashids-encoded token

These tokens are regenerated for each API request to prevent replay attacks.
